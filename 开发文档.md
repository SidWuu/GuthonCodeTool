# 谷神源码 Hub 只读同步与项目 Effective 调用索引开发文档

## 1. 目标

实现一个本地只读工具，从谷神开发平台数据库同步已签入源码，按产品层和项目层组织源码镜像，并为每个项目生成最终生效的 effective 源码视图和调用索引。

核心目标：

- 谷神开发平台数据库是唯一真实源码来源。
- 本地只维护一个 `gusen-code-hub`，项目目录只引用 Hub，不重复保存全量源码。
- 所有环境都使用开发平台源码，不再设计开发、测试、生产环境源码层。
- 工具只读数据库，禁止直接回写源码表。
- 修改源码时只生成工作副本，由人工复制回谷神开发平台保存、提交、签入。

## 2. 分层口径

```text
谷神开发平台数据库
  ├── PRODUCT 产品源码层
  └── PROJECT 项目源码层

项目 EFFECTIVE 源码 = PROJECT 优先 + PRODUCT 兜底
```

覆盖规则：

```text
如果项目层存在同一源码对象：使用 PROJECT
否则：使用 PRODUCT
```

第一阶段不做环境层：

```text
无 dev-source
无 test-source
无 prod-source
```

## 5. 第一阶段源码表

只实现两类源码对象：

### 5.1 页面源码 `page`

物理表名和字段名不写入公开文档，统一由 `config/source-tables.yaml` 配置。页面源码至少需要映射这些语义字段：

```text
page id
page alias
page name
module id
module name
system id
page content
update time
check-out user
check-out time
check-in time
version marker
error flag
error message
```

页面对象主键：

```text
page id
```

页面 effective 覆盖键：

```text
source_table + page alias
```

风险：当前库可能存在少量重复页面别名。本地目录必须带页面 ID，避免覆盖。

```text
var/source/readonly/products/{system-name}/page/{module-name}/{page-name} {page-id}/
var/source/readonly/project/{project.name}/{system-name}/page/{module-name}/{page-name} {page-id}/
var/source/effective/project/{project.name}/{system-name}/page/{module-name}/{page-name} {page-id}/
```

如果同一层同一页面别名出现多条记录，状态标记为 `DUPLICATE_ALIAS`，不静默覆盖。

### 5.2 过程函数源码 `procedure`

物理表名和字段名由 `config/source-tables.yaml` 配置。过程函数源码至少需要映射这些语义字段：

```text
procedure id
procedure alias
procedure name
data source id
function id
function name
function params
function script
update time
check-out user
check-out time
check-in time
version marker
error flag
error message
```

过程函数对象主键：

```text
procedure id + function id
```

过程函数 effective 覆盖键：

```text
source_table + procedure alias + function id
```

本地目录：

```text
var/source/readonly/products/{system-name}/procedure/{procedure-alias}/{fun-id}/
var/source/readonly/project/{project.name}/{system-name}/procedure/{procedure-alias}/{fun-id}/
var/source/effective/project/{project.name}/{system-name}/procedure/{procedure-alias}/{fun-id}/
```

过程函数归属规则：

- 一个子系统不会属于多个数据源。
- 多个子系统可能共用同一个数据源 ID。
- 如果数据源 ID 只对应一个已配置子系统，过程函数根目录使用该子系统名称。
- 如果数据源 ID 对应多个已配置子系统，过程函数只存一份，根目录使用 `systems.yaml` 中第一个匹配子系统的名称，其余子系统的 `procedure` 目录链接到第一个目录。

## 6. 同步过滤

当前开发对象由 `config/sync.yaml` 控制：

```yaml
sync:
  ACTIVE: products.demo-product
```

可选格式：

```text
products.{product_id}
projects.{project_id}
```

全量拉取和定时拉取都只同步 `ACTIVE` 指向的产品或项目源码。`ACTIVE=projects.{project_id}` 时，只重建该项目的 effective 目录；产品兜底源码来自本地已有产品索引，不额外连接产品库。

硬规则：

```text
签入时间为空：跳过，不同步，不索引。
签入时间非空：允许同步。
```

签入时间非空只表示可作为已提交源码参与同步，不表示当前无人签出。查库发现存在已签入但签出用户不为空的记录，因此必须把签出信息写入 `meta.json` 和索引：

```text
check_out_user_id
check_out_date
checkout_status = CHECKED_OUT | NOT_CHECKED_OUT
```

错误版本过滤：

```sql
按配置的错误标记字段过滤错误版本。
```

空源码处理：

```text
源码内容为空：生成 meta.json，status = EMPTY_CONTENT，不生成调用索引。
```

## 7. 增量查询

同步任务每次使用回看窗口：

```text
sync_from_time = last_success_time - lookback_minutes
```

默认：

```text
lookback_minutes = 10
```

页面和过程函数增量 SQL 由 `config/source-tables.yaml` 生成。公开仓库只保留逻辑类型 `page` / `procedure`，不提交物理表名和字段名。

变更标识：

```text
if version marker is not null:
    change_key = 'VERSION:' + version marker
else if check-in time is not null:
    change_key = 'CHECK_IN:' + check-in time
else:
    change_key = 'UPDATE:' + update time
```

页面和过程函数的更新时间字段都来自 `source-tables.yaml`。

## 9. PRODUCT / PROJECT 归属

当前口径：项目继承产品代码做定制化开发，页面别名和过程函数别名可能完全相同。PRODUCT / PROJECT 不靠别名前缀区分，靠数据源区分：

```text
产品开发平台数据库 -> PRODUCT
项目开发平台数据库 -> PROJECT
```

示例：

```yaml
datasource:
  product-dev:
    database: ${GUSEN_PRODUCT_DB_NAME}
  project-dev:
    database: ${GUSEN_PROJECT_DB_NAME}

products:
  demo-product:
    datasource: product-dev
    include:
      all: true

projects:
  demo-project:
    datasource: project-dev
    product_id: demo-product
    include:
      all: true
```

effective 构建时，如果 PRODUCT 和 PROJECT 存在同一覆盖键：

```text
source_table + page alias
source_table + procedure alias + function id
```

则选择 PROJECT，产品版本写入 `override-index.md` 作为被覆盖版本。

## 10. 子系统过滤

如果只需要同步某个子系统，把系统数据 JSON 放到 `config/system-data.json`，再配置 `config/systems.yaml`：

```yaml
systems:
  data_file: system-data.json
  include:
    system_codes:
      - DEMO_SYSTEM
```

系统数据 JSON 支持数组或带 `data / rows / records / items / children / list` 的对象结构。工具会递归查找这些字段：

```text
system code
system id
system alias
data source id
```

过滤规则：

```text
page：按 `source-tables.yaml` 配置的页面子系统字段过滤。
procedure：如果系统数据能解析到数据源 ID，则按 `source-tables.yaml` 配置的数据源字段过滤。
```

如果系统数据没有过程函数对应的数据源 ID，第一阶段不额外过滤过程函数，避免误删共享过程函数。

## 12. 解析规则

页面 `XML` 是 JSON 字符串：

```text
1. 读取 XML。
2. JSON 反序列化。
3. 如果结果仍是字符串，再反序列化一次。
4. 递归扫描脚本字段。
5. 拆分保存到 scripts/。
```

脚本字段：

```text
beforeSaveScript
afterSaveScript
onClickScript
onOpenScript
onCreateScript
onAfterLoadScript
onChangeScript
onBeforeWinCloseScript
doMethodScript
sql
```

过程函数：

```text
procedure script -> source.vm
function params -> meta.json 和 README.md
```

## 13. 调用识别

第一阶段只识别高价值静态调用：

```text
$vs.proc.invoke("alias", "fun", ...)
$vs.proc.find("alias") + $proc.fun(...)
$vm.open("pageAlias", ...)
$vm.openDialog("pageAlias", ...)
gUtil.request("serviceAlias", ...)
```

动态调用不丢弃，写入低置信动态调用点：

```text
$vs.proc.invoke(procAlias, funId, param)
$proc.$methodName(param)
```

输出：

```text
var/knowledge/projects/{project_id}/invoke-index.md
var/knowledge/projects/{project_id}/invoke-callers.md
var/knowledge/projects/{project_id}/invoke-callees.md
var/knowledge/projects/{project_id}/dynamic-invoke-points.md
```

调用索引只是辅助证据，不声明为完整调用图。

## 14. 工作副本

修改源码时只允许生成工作副本。产品源码从 `source/readonly` 生成，项目源码从 `source/effective` 生成。

```text
var/source/workcopy/products/{product.name}/{yyyyMMdd-HHmmss}-{alias-or-fun}/
var/source/workcopy/{project.name}/{yyyyMMdd-HHmmss}-{alias-or-fun}/
```

产品命令：

```bash
python scripts/create_work_copy.py \
  --product demo-product \
  --type procedure \
  --alias demo.procedure.common \
  --fun checkDetailState
```

项目命令：

```bash
python scripts/create_work_copy.py \
  --project demo-project \
  --type page \
  --alias demo.page.editor
```

```bash
python scripts/create_work_copy.py \
  --project demo-project \
  --type procedure \
  --alias demo.procedure.common \
  --fun checkDetailState
```

工具不回写数据库。人工将工作副本修改结果复制回谷神开发平台。

## 17. 最终原则

```text
谷神开发平台数据库 = 唯一真实源码源头
环境源码层 = 不存在
PRODUCT = 产品基线源码
PROJECT = 项目定制源码
EFFECTIVE = 项目最终生效源码
var/source/readonly = 已签入源码只读镜像
var/source/effective = 项目生效源码只读视图
var/source/workcopy = 临时修改副本
var/runtime/index/hub.db = 唯一运行索引
var/knowledge = 人工全量浏览与 AI 查询入口说明
工具 = 只读同步、解析、索引、导出
人工 = 唯一回写谷神开发平台的方式
```
