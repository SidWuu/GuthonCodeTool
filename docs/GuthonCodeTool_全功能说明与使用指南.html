1. 工具简介

GuthonCodeTool 用于把谷神开发平台中的页面源码、过程函数、数据库表结构、单据类型和调用关系整理到本地，形成可检索、可比较、可交付、可供 AI 辅助分析的开发工作区。

它主要解决以下问题：

  • 平台源码分散在数据库和在线编辑器中，不便于全文检索和版本比较。
  • 页面、过程函数之间存在调用关系，人工查找效率低。
  • 本地修改可能与平台最新版本分叉，直接覆盖存在风险。
  • 数据表字段、单据类型等元数据需要反复进入平台查询。
  • 在线开发页面缺少快捷跳转、字段复制、源码快速拉取等辅助能力。
  • AI 编程工具无法直接理解平台数据库中的源码和结构。

工具采用“根仓库保存工具代码，var/ 保存本机私有数据”的模式。源码、数据库结构、SQLite 索引、日志和工作副本不会提交到公开工具仓库。

2. 功能总览

2.1 源码 Hub

源码 Hub 是工具的核心能力，负责从谷神源码表同步页面和过程函数源码。

主要能力：

  • 同步产品层源码和项目层源码，二者相互隔离。
  • 通过 sync.ACTIVE 指定当前开发对象。
  • 每个产品或项目使用独立 SQLite 索引库，避免索引相互污染。
  • 按子系统别名和数据源范围过滤。
  • 默认同步已签入源码。
  • 可配置允许同步指定签出人的未签入源码。
  • 通过 VERSION_MAC 优先判断源码是否变化。
  • 未变化时复用本地只读镜像，不重复写文件和重建索引。
  • 变化时更新本地文件并重新生成调用关系。
  • 支持初始化索引、一次性全量或增量同步、手动拉取单个对象。

适用场景：

  • 将谷神平台源码定期同步到本地。
  • 使用 VS Code、IDEA、Codex、Claude Code 等工具分析业务代码。
  • 查询某个页面或过程函数的来源、调用方和被调用方。
  • 在不同产品、项目之间切换开发上下文。

2.2 页面源码导出

页面源码会按业务层级自动整理目录，典型结构为：

text
{子系统}/page/{模型序号}_{模型}/{模块序号}_{模块}/{页面名 页面ID}

目录中的常见文件：

text
meta.json
raw.json
scripts/*.js
scripts/*.sql
scripts/*.vm
README.md

具体能力：

  • 保存页面完整原始 JSON 为 raw.json。
  • 将页面脚本、SQL、服务组件脚本拆分到 scripts/。
  • 模型和模块序号统一补齐三位，便于自然排序。
  • 根据模型父子链组织页面目录。
  • 项目 PAGE 后台脚本存在独立 @inherit(); 标记时，自动使用产品快照 superScript。
  • serviceEvents、pageEvents 的继承标记分别替换为产品快照。
  • 替换后为空的事件脚本不生成文件。
  • 服务组件 compScript 即使为空，也生成空的 compScript.vm。
  • 手动拉取页面时只处理当前页面。
  • 目录规则变化后可通过强制刷新重建路径并清理旧目录。

2.3 过程函数源码导出

过程函数目录结构：

text
{子系统}/procedure/{过程别名}/{函数名}

目录中的常见文件：

text
meta.json
source.vm
README.md

具体能力：

  • 将过程函数源码保存为 source.vm。
  • 支持项目层过程函数继承产品快照。
  • 独立 @inherit(); 标记行会替换为配置字段映射的产品脚本。
  • 多个子系统共用同一数据源时，源码只保存一份。
  • 其他子系统目录通过链接指向同一份源码，减少重复文件。

2.4 工作副本

只读镜像位于：

text
var/source/readonly/

开发工作副本位于：

text
var/source/workcopy/

工作副本用于本地修改和人工回写平台，包含：

text
.guthon-baseline/
source-meta.json
diff.md
delivery.md

主要能力：

  • 从只读源码生成可编辑工作副本。
  • 保存创建工作副本时的基线版本。
  • 自动比较本地修改与上游版本。
  • 生成 Markdown 差异文件 diff.md。
  • 生成交付清单 delivery.md。
  • 本地改动已人工回写平台后，自动识别并刷新基线。
  • 本地和平台同时变化且内容分叉时，拒绝覆盖并保留本地修改。
  • 支持以下状态：

- CLEAN：本地与基线一致。

- LOCAL_CHANGED：仅本地修改。

- UPSTREAM_CHANGED：仅上游修改。

- CONFLICT：本地与上游均修改且分叉。

- UPSTREAM_MISSING：上游对象不存在。

重要说明：工具不会自动将本地文件写回谷神平台。开发人员仍需人工复制到平台、保存、提交和签入。

2.5 SQLite 源码索引与调用关系

工具为每个 sync.ACTIVE 建立独立 SQLite 数据库，典型位置：

text
var/runtime/index/products/<product_id>.db
var/runtime/index/projects/<project_id>.db

索引内容包括：

  • 源码对象基础信息。
  • 本地源码路径。
  • 版本号、更新时间、签出人、签入状态。
  • 静态过程函数调用关系。
  • 调用所在脚本、JSON 路径和行号。
  • 被调用过程别名和方法名。
  • 调用类型和置信度。
  • 无法静态确认的动态调用点及原因。
  • 同步状态。

主要用途：

  • 按名称、ID、别名搜索源码。
  • 查询某个过程函数调用了哪些函数。
  • 查询某个过程函数被哪些页面或函数调用。
  • 为 AI 助手提供局部上下文，避免每次加载全部源码。
  • 定位低置信度动态调用，辅助人工检查。

默认只生成轻量状态入口。需要完整 Markdown 知识索引时，可显式运行：

bash
.venv/bin/python scripts/export_hub_markdown.py

2.6 数据库表结构导出

脚本：

bash
.venv/bin/python scripts/export_table_schema_sql.py

数据来源：

  • gd_tables
  • gd_tables_field
  • gd_system

输出目录：

text
var/database/schema/

主要能力：

  • 默认使用 sync.ACTIVE 对应的数据源。
  • 根据 sync.yaml 中配置的子系统别名自动解析数据源 ID。
  • 支持导出当前范围的全部表。
  • 支持临时指定一个或多个数据源。
  • 浏览器扩展中可只导出当前选中的表。
  • 输出精简 JSON，省略空值字段。

临时覆盖数据源示例：

bash
.venv/bin/python scripts/export_table_schema_sql.py --data-source-ids 0015,0018

保留的表级字段包括：

text
tableId, tableName, dataSourceId, systemName, systemAliasId,
cacheType, cacheKey, cacheDataField, fields

保留的字段级信息包括：

text
fieldId, fieldName, dataType, dataLength, dataPrecision,
isPrimary, isCanNull, isIncrement, defaultValue, fieldRemark,
dataAuthField, isCipher, orderNo

2.8 Guthon Bridge 浏览器扩展

GuthonBridge 是面向谷神在线开发页面的浏览器端开发增强工具，由 Chrome 扩展和本地 HTTP Bridge 服务组成。它把在线开发页面与 Guthon Hub、本地源码目录和数据库结构导出工具连接起来，在不直接修改平台源码的前提下，为日常开发补充导航、拉取、复制和字段迁移能力。

主要价值减少在谷神平台中反复查找、复制和切换页面的操作。
运行方式Chrome 扩展识别页面上下文，本地 Bridge 调用脚本并保存文件。
安全边界不自动回推源码，不自动签出页面或过程函数。

核心功能

  • 过程函数快捷跳转：在过程函数和模块后台脚本中，按住 Ctrl，macOS 使用 Cmd，点击高亮的静态方法名即可打开目标过程函数；也支持当前过程函数内部的子方法跳转。
  • 长脚本函数定位：配合谷神自带的定位按钮，在高亮函数名后自动滚动到对应过程函数位置。
  • 当前页面源码保存：识别过程函数页面中的当前脚本，将在线编辑内容直接保存为本地文件，并记录远端对象、本地路径和版本元数据。
  • 源码表版本拉取：在过程函数页或模块开发页调用 Guthon Hub，从源码表拉取已保存版本，更新 readonly 镜像并生成工作副本。
  • 工作副本冲突保护：本地修改已经回写上游时刷新基线；本地和上游内容分叉时拒绝覆盖，保留本地文件并明确提示冲突。
  • 表结构按需导出：在数据表管理页识别当前数据源；未选择表时导出全部表结构，选择后仅导出指定表。
  • 单据类型按需导出：在单据类型页识别当前数据源和选中项，可导出全部或指定单据类型。
  • 页面字段复制模式：在模块开发页按页面块展示字段结构，生成便于复制、整理和交给 AI 分析的纯文本。
  • 组件字段平移：复制输入组件字段或表格组件列，再粘贴到另一个已选组件；按 fieldId 自动跳过重复项。
  • 页面悬浮入口：在支持的开发页面提供可拖动悬浮按钮，常用操作无需频繁打开扩展弹窗。

支持的调用跳转形式

java
$vs.proc.invoke("过程别名", "函数名", ...)
#set($proc = $vs.proc.find("过程别名"))
$proc.函数名(...)

只有过程别名和函数名为固定字符串时才能可靠跳转，动态包名和动态方法名不在当前支持范围内。

启动与安装

bash
cd plugins/GuthonBridge
npm run start:bridge

本地 Bridge 默认监听 127.0.0.1:17361，健康检查地址为 http://127.0.0.1:17361/health。Chrome 扩展通过开发者模式加载 plugins/GuthonBridge/extension/。

允许访问的谷神平台地址在 extension/host-config.js 中配置，可限制协议、IPv4 CIDR、域名后缀和路径前缀。

限制与安全设计

  • 不支持本地文件自动回推开发平台。
  • 不支持自动签出过程函数或 PAGE。
  • 收到签出或回推命令时直接返回错误，不调用平台保存接口。
  • 复制模式依赖谷神页面 DOM 和 Vue 实例结构，平台页面结构较大变化时需要同步适配。
  • 当前源码保存目录需要手工输入绝对路径，同名文件会覆盖。

2.9 GuthonVSCodeExtension VS Code 扩展

GuthonVSCodeExtension 是面向谷神方言脚本的本地 VS Code 扩展，目前提供两类核心能力：离线代码补全，以及从过程函数调用跳转到本地源码。扩展不依赖 IntelliCode，不调用 Copilot，补全数据随 VSIX 安装包离线提供。

适用文件Java 后端脚本、JavaScript 页面脚本和 SQL 方言脚本。
配套场景适合搭配 Guthon Hub 导出的 readonly 源码和 workcopy 工作副本。
隐私特性补全数据完全本地化，不需要将业务代码发送到外部服务。

离线代码补全

  • 按语言自动选择数据：Java 文件使用 Java API 数据，JavaScript 文件使用 JavaScript API 数据，SQL 文件使用 SQL API 数据。
  • 跨数据源路由：Java 文件中输入 sqltools 补全 SQLTools.*；输入 sql 补全 $vs.sqlTools.*；输入 sqlh 补全 $vs.sqlHelper.*。
  • 谷神基础语法片段：提供 set、if、ifelse、foreach、while、continue、break、tryCatchFinally 和 function。
  • 完整提示说明：候选列表显示补全前缀和简短说明，右侧详情面板展示插入内容和完整 API 描述。
  • 支持临时文件：未保存文件只需将语言模式切换为 Java、JavaScript 或 SQL，也可以使用补全。

本地过程函数源码跳转

在包含 Guthon Hub 本地源码的 VS Code 工作区中，可以从 Java 后端脚本的过程函数调用直接跳转到对应源码。

java
$vs.proc.invoke("过程别名", "函数名", $参数)
#set($proc = $vs.proc.find("过程别名"))
$proc.函数名($参数)

将光标放在函数名上，通过 macOS 的 Cmd+Click、Windows/Linux 的 Ctrl+Click，或按 F12 执行“转到定义”。当前仅支持过程别名和函数名为固定字符串的调用。

安装方式

从 GuthonCodeTool Releases 下载 gushen-vscode-completion-0.1.0.vsix：

bash
code --install-extension /path/to/gushen-vscode-completion-0.1.0.vsix --force

安装完成后执行 Developer: Reload Window；也可以在扩展面板中使用 Install from VSIX...。

规则与补全数据维护

  • rules.json：定义默认语言数据源和特殊触发词路由；也可通过 gushenCompletion.rulesPath 指定自定义绝对路径。
  • data/manual.json：维护人工编写的谷神基础语法片段，重新生成 API 数据时不会被覆盖。
  • data/java.json、data/javascript.json、data/sql.json、data/index.json：由 API 文档生成的离线补全数据。
bash
cd plugins/GuthonVSCodeExtension/gushen-vscode-completion
npm run build:data -- /path/to/api-docs
npm test
vsce package
code --install-extension gushen-vscode-completion-0.1.0.vsix --force

API 文档目录需要包含 java.md、javascript.md 和 sql.md。重新生成数据不会覆盖 rules.json 和 data/manual.json。

4. 快速安装

4.1 获取代码

bash
git clone https://github.com/SidWuu/GuthonCodeTool.git
cd GuthonCodeTool

4.2 创建 Python 环境

bash
python3 -m venv .venv
source .venv/bin/activate

根据仓库依赖安装 Python 包。若未安装 PyYAML，工具内置轻量 YAML 解析器可处理当前模板，但建议在正式环境安装完整依赖。

4.3 创建配置文件

bash
cp config/example/datasource.example.yaml config/datasource.yaml
cp config/example/products.example.yaml config/products.yaml
cp config/example/projects.example.yaml config/projects.yaml
cp config/example/source-tables.example.yaml config/source-tables.yaml
cp config/example/sync.example.yaml config/sync.yaml

需要填写：

  • 产品库和项目库连接。
  • 产品定义。
  • 项目定义。
  • 谷神源码表和字段映射。
  • 当前开发对象 sync.ACTIVE。
  • 需要同步的子系统别名。
  • 允许读取的未签入源码签出人。

关键示例：

yaml
sync:
  ACTIVE: products.demo-product

systems:
  include:
    system_aliases:
      - demo.system

rules:
  allow_unchecked_check_out_user_ids:
    - U00000XXXX

切换到项目开发：

yaml
sync:
  ACTIVE: projects.demo-project

ACTIVE 会同时决定：

  • 同步哪个产品或项目。
  • 使用哪个 SQLite 索引库。
  • 工作副本从哪个索引读取。
  • 表结构和单据类型默认连接哪个数据库。

7. 推荐工作流

7.1 日常开发

  1. 设置当前 sync.ACTIVE。
  2. 执行源码同步。
  3. 从索引或浏览器扩展创建工作副本。
  4. 在本地编辑器或 AI 工具中分析和修改。
  5. 使用 workcopy.py diff 检查变化。
  6. 将修改人工复制回谷神平台。
  7. 在平台保存、提交、签入。
  8. 再次拉取源码。
  9. 工具确认上游已包含本地改动后刷新基线。
  10. 使用 workcopy.py package 生成交付清单。

7.2 分析陌生业务

  1. 同步对应产品或项目。
  2. 导出数据库表结构和单据类型。
  3. 使用 SQLite 索引搜索页面、过程函数和调用链。
  4. 将局部源码、调用关系、表结构提供给 AI。
  5. 对低置信动态调用点进行人工确认。

7.3 页面字段迁移

  1. 在模块开发页选择来源组件。
  2. 点击“复制字段”。
  3. 勾选需要复制的字段。
  4. 选择目标组件。
  5. 点击“粘贴字段”。
  6. 检查重复字段和组件类型。
  7. 人工保存页面。

8. 使用边界与注意事项

  • 根仓库可以公开，但 var/、数据库连接配置和缓存不得提交。
  • datasource.yaml 和 system-data.json 应保持本地私有。
  • 工作副本冲突时不要强制覆盖，应先比较本地和上游源码。
  • VERSION_MAC 是主要变更判断依据，平台数据异常时应结合更新时间核对。
  • 未签入源码只有签出人位于允许名单时才会读取。
  • 多个子系统共用数据源时，过程函数目录可能是链接，不要误认为源码缺失。
  • 浏览器扩展依赖谷神页面 DOM 和 Vue 实例结构，平台大改后可能需要适配。
  • 复杂 tab、form、table 组合的字段复制兼容性仍需实际检查。
  • 浏览器当前源码保存会覆盖同名文件。
  • 修改扩展代码后，需要重新加载扩展并刷新平台页面。
  • Bridge 修改端口后，扩展访问地址和权限也需要同步调整。
  • 工具不会自动回写、签出或签入源码，这是安全设计，不是功能缺失。
